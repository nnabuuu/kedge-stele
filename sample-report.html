Decision Archive · ccaas runtime · agent_files bootstrap · 2026-06-03

ccaas session bootstrap 接上 entity-scoped agent_files — 新 session bind 同一 sourceIdentity 后,agent fs 里能直接 Read/Edit 历史附件。

这是什么。 Session bootstrap 阶段新加一道 SessionFileMaterializer:把 agent_files 表里属于当前 session 所绑 entity 的文件,按 file.originalPath 复制进 session 的 workspace。 调用走 MessageWorker 的直接调用链(在 assetSyncer.sync 之后,orchestrateMessage 之前),同时挂 @OnEvent('session.bound') 作 safety net。

为什么做。 上一轮 PR (6eacf156) 把 agent_files 升级到了 entity-scoped: 文件能跨 session 通过 entity 关联了。但"entity 的存在感"只到了 DB + REST 层—— 新建 session bind 同一个 sourceIdentity 时,agent 的 fs 里依旧是空的,只能通过 /api/v1/files/:id/download 拿。这跟用户心智模型 ("这个课程项目的所有内容") 不一致。 用户原话: "workspace 下面所有的内容都要被一起导入 reconstruct 成为 claude 的 fs 的一部分可以供修改,对吧?" —— 这次 PR 把"对吧"变成"是"。

范围。 In: SessionFileMaterializer service + MessageWorker 直接调用 + 事件 safety net + idempotency (metadata flag + in-flight Promise dedup) + caps (500 文件 / 50 MB) + path-safety.ts helper 抽取。 Out: agentfs base 扩出 entity 共享层 (多 session 重复占 delta 空间)、cascade DELETE on entity removal、Prometheus metrics、cap 命中后的 UX 信号。

D-01

问: 给 agent_files 也加 bootstrap,还是接受 gap?

Runtime · Scope
触发 用户直接点出: "workspace 下面所有的内容都要被一起导入 reconstruct 成为 claude 的 fs 的一部分可以供修改,对吧?" —— 调研发现 artifact bootstrap 已完整,agent_files 这一侧没接上。 约束 ccaas 现有的 bootstrap 链路 (SessionAssetSyncer) 只覆盖 solution 拥有的 artifacts; agent_files 是平台拥有的,没人来塞进 fs。
Option方案Verdict
A 接受 gap — 历史 entity files 只能通过 /files/:id/download 拿到,agent 在 fs 里看不见 ❌ 拒 — entity-scoped 的语义不完整,只有 DB 层的存在感。
B 新增 SessionFileMaterializer,session bootstrap 时按 originalPath 复制进 workspace ✓ 选 — entity 的"存在感"延伸到 fs 层。
为什么这么选

上一轮 PR 让 ccaas 知道文件属于谁,但 agent 在自己的 fs 里依然看不见 —— "知道"和"能用"在 agent runtime 这个语境里是两件事。 DB-only 的 entity 关系只对 ccaas 的记账有用,对 agent 没有用 —— agent 不会主动调 /files/... download API,它读 fs。

要让 entity-scoped 真正生效,必须把内容投影回 agent 看得到的世界。这件事 ccaas 已经为 solution-owned artifacts 做了, 直接沿用同一套架构 hook 一遍而不是另发明体系,是成本最低的对齐方式。

Locked in
双触发(worker 直接调用 + event safety net);session.bound 后 entity files 必须在 agent 看见之前就位。
Locked out
"cross-session 文件复用"现在是 expectation —— 下游 lifecycle (entity cascade delete) 必须跟着 entity 走,session 单独清掉对 entity 影响为 0。

D-02

问: 触发点是 session.bound 事件,还是 MessageWorker 直接调用链?

Backend · Timing
触发 第一版 plan 写"挂 @OnEvent('session.bound')",用户反应"还是有点不保险",要求再深挖。Deep-dig 发现事件派发是 fire-and-forget microtask,跟 worker 的sync → spawn CLI同步链时序不可控: 事件 handler 可能 spawn 之后才跑完,导致 agent 第一轮看到空 fs。 约束 SessionAssetSyncer 已经踩过这个坑 (G4 fix); 它现在的解法是 worker await assetSyncer.sync(sessionId) 直接调用,不依赖事件。新的 materializer 应当跟进。
Option方案Verdict
A 只挂 @OnEvent('session.bound') 事件 handler ❌ 拒 — agent 可能在 fs 物化完成前就 spawn 跑。
B MessageWorker 直接调用 await materialize() + 事件 handler 作 safety net ✓ 选 — 主路径同步保序,事件兜底非 worker 路径。
C 只直接调,完全不挂事件 ❌ 拒 — admin 工具/未来路径不走 worker 时漏掉。
为什么这么选

NestJS EventEmitter2.emit 是同步 return + 异步派发: emit 调用立即返回,handler 走 microtask queue,跟 worker 的下一行同步代码(orchestrateMessage)赛跑。 microtask 的赢面看 V8 调度细节,不能假定。这是一类整个 ccaas 都已经遇到过的问题—— SessionAssetSyncer 当时的 G4 fix 已经把"关键路径不依赖事件"立成了 ccaas 的范式。

但完全不挂事件又会漏掉非 worker 路径 (admin 调试、未来其他流程) 的 attach。所以选 双触发: 主路径直接 await 保证正确性,事件作为兜底。 代价是同一个 session 可能被两条路径同时打到 —— 这是后面 D-04 idempotency 要解决的问题。

Locked in
MessageWorker 显式注入 SessionFileMaterializer 依赖;关键路径不再相信 event 派发顺序。
Locked out
"事件丢了不影响功能"成为 ccaas 的契约 —— 将来不能用事件作为唯一的状态机驱动力。

D-03

问: Agent 改了一个物化进来的文件,语义是什么?

Design · Semantics
触发 Plan 阶段问"agent 改了物化文件怎么办",用户答: "这里我们不是用 agentFS 直接完成的吗?" 约束 反向 sync(改了就写回原 storedPath)需要一整套 syncer 架构 + 冲突解决,工作量与 WorkspaceArtifactSource 相当;但 agentfs 已经提供了 delta overlay。
Option方案Verdict
A 反向 sync,agent 改 → 写回 file.storedPath ❌ 拒 — 跨进程 + 冲突,工程量大。
B 进来只读,修改只生新行 — 复用 agentfs delta + write-file-tracker ✓ 选 — 零新代码,语义跟现有 agent_files 一致。
C 不物化,只在 system prompt 提示 "用 download tool 拿" ❌ 拒 — agent 不能 grep/edit,功能上残废。
为什么这么选

这个决定整个由 ccaas 既有的两件事拼起来,没有新代码:

(1) agentfs delta overlay: NestJS service 在 mount 点 fs.copyFile 写入的字节落进per-session SQLite delta(deep-dig 确认过), agent 之后 Edit 这个文件也落进同一个 delta —— 原 storedPath(持久化目录)永不被动。

(2) local provider 上的 write-file-tracker: agent 改文件会被 hook 捕获,创建新的 agent_file 行,继承相同的 sourceIdentity。 原行 + 持久化副本也是不被动。

两个 provider 上的语义都自然满足"进来只读 + 修改只生新行" —— 语义不需要我设计,既有 invariants 自动提供。

Locked in
"agent_files 只增不改"成为 ccaas 的不变量,后续 versioning / audit 设计都可以依赖。
Locked out
agentfs delta 占用按 session 累积 —— 多 session 同 entity 时每个都复制一份;DEF-01 留作 Phase 2 优化。

D-04

问: Worker 直接调用 + 事件 handler 同 session 跑 → race 怎么挡?

Concurrency · Race
触发 Round-1 code review 找到: attachWorkspaceSource 同步 emit session.bound,worker 的 await 接着跑 — emit 返回后 event handler 还没跑完。Worker 直接调和 event handler 都能通过 files.bootstrapped metadata flag 检查 → 双方都开始 copy → 双倍 IO,可能虚假触发 cap。 约束 metadata flag 是 await 异步写,从读到写之间有窗口 — flag 单独防不住 intra-process race。
Option方案Verdict
A 只 metadata flag (round-1 版本) ❌ 拒 — race window 真实存在。
B metadata flag (跨重启) + 进程内 Map<sessionId, Promise> single-flight ✓ 选 — 两个层次互补。
C DB-level lock (advisory lock / row lock) ❌ 拒 — 大锤,跨进程在 ccaas 不需要(SQLite + single instance)。
为什么这么选

关键洞察: race 不是跨进程的,是同进程的。ccaas 当前是 single-replica;worker 和 event handler 是同一个 Node 实例的两个调用链。 这种 race 用进程内的 Promise Map 就能干净挡住 —— map.get + map.set 在 V8 里是同步的、原子的, 两个调用链中后到的一个看到的就是前者放进去的 Promise,直接 await 同一个执行结果。

metadata flag 留着是为了跨重启 idempotency —— 进程挂了重起来,之前已经 bootstrap 过的 session 不要再 bootstrap。两个层次解决两个问题,各司其职。

DB lock 会拉起 cross-process 的复杂度 —— 当前 ccaas 没这个 scenario,把锁机制画得太大反而是 over-design。当多实例真出现时再加 (OQ-01)。

Locked in
"single-process race 用 Promise Map 解决"成为 ccaas 范式;后续类似 race 可以同样套。
Locked out
多实例部署时需要重新审视 race protection(OQ-01)。

D-05

问: 物化进 workspace 时放哪个路径?

Design · Layout
触发 Plan 决策点。Upload 时是 images/cover.png,物化时也要放在那个语义点上,不然 agent 拿不准位置。 约束 <workspace>/artifacts/ 已经被 solution-owned 内容占了;路径冲突时要有规则。
Option方案Verdict
A 按 file.originalPath 原路径还原 — 冲突时 artifacts 优先 ✓ 选 — agent 看到的路径跟 upload 时一致。
B 统一放进 <workspace>/uploads/ 子目录 ❌ 拒 — 跟 attach-file MCP 文档里"在 outputs/ 下拿音频"语义脱节。
C 同路径顺序覆盖,materializer 比 artifacts 后写 ❌ 拒 — silent overwrite,solution 拥有的内容被平台拥有的覆盖,trust 反向了。
为什么这么选

originalPath 是 agent 当初看到的位置 —— upload 时是 images/x.png,attach-file MCP 文档里写"agent 应当去 outputs/ 拿生成的音频", 这些语义都预设了相对路径稳定。换一个统一子目录会破坏这层契约。

冲突时 artifacts 赢 —— 因为solution 是 source of truth,平台是 derived: solution 拥有的内容定义了项目;agent_files 是基于那个项目衍生出来的产物。 冲突时该让"定义"覆盖"产物",反过来就是derived 数据污染 source-of-truth,长期会乱。

Locked in
workspace 命名空间是共享的;artifacts vs agent_files 第一个落地的赢。
Locked out
完全无冲突的隔离模型做不到 —— 但实际冲突概率低(artifacts 一般是项目源文件,agent_files 是用户上传/agent 产出)。

D-06

问: path-safety helper 怎么共享给 materializer?

Backend · Refactor
触发 SessionFileMaterializer 也需要 resolveInsideWorkspace 防 originalPath 被改坏后逃出 workspace。这个 helper 之前是 FilesService 的 private method。 约束 helper 是纯函数 + 用 module-level Logger,本来就没 DI 依赖;不该为了共享一段 path 计算把 materializer 绑到 FilesService 上。
Option方案Verdict
A FilesService.resolveInsideWorkspace 改 public,materializer 注入 FilesService ❌ 拒 — 引入一条只为 path 计算存在的 DI 依赖。
B 复制粘贴,两份实现 ❌ 拒 — 已经被 round-1/round-2 review 锤过的安全代码,容忍不了 drift。
C 抽到独立 files/path-safety.ts module — 零 DI 纯函数 + module-level Logger ✓ 选 — 第二 consumer 自然落地,既有测试一并搬到 path-safety.spec.ts。
为什么这么选

这段代码已经经历了两轮 code review 的迭代(C1 path-traversal critical、round-2 item 1 symlink TOCTOU、prefix-sibling 防御), 是 ccaas 整个 file 流里安全最敏感的一段。复制粘贴等于给 drift 开口子,六个月后两份会变得不一样。

通过 FilesService 暴露(option A)又会拖来一条 DI 边 —— materializer 不需要 FilesService 的任何其他能力,只需要这两个工具函数。 对应 "DI 边应该按真实依赖画" 的原则,不该用 DI 容器传递无状态工具。

独立 module 是最低成本的扁平化: 19 个测试集中到一个 path-safety.spec.ts,后续添 consumer 不需要再决策。

Locked in
resolveInsideWorkspace/safeFsSegment 成为正式公共 surface,改它需要考虑 backwards-compat。
Locked out
原"就近修改"的灵活性没了 — 改 helper 要顾两个 consumer。但代价低,因为接口稳定。

02 —

Deferred 这次显式没做的

5 项 · 触发条件下重新审视
DEF-01

agentfs base 扩出 entity 共享层

现状每个 session 物化时把 entity 文件复制进自己的 delta — 多 session 同 entity 时 delta 空间按 session 数线性占。一个 100 MB 课件 × 10 个并发 session = 1 GB delta。

推迟理由改 BaseMaterializer + mount 配置以支持 _agentfs_base/tenants/{solutionId}/entities/{sourceIdentity}/... 共享层,工作量大;当前数据量没到瓶颈。

重新审视的触发单 entity 并发 session 数 × 课件大小成为磁盘告警源。Profiling 显示 delta 占用是主要成本时。

DEF-02

Entity cascade DELETE

现状solution backend 删 project 时 ccaas 不知道; agent_files 行 + 持久化副本永远留着。

推迟理由需要 solution 调一个新协议 DELETE /workspaces/:sourceIdentity/files,涉及 solution 侧改动;当前没有 solution 真的删 project + 报告 stale files。

重新审视的触发第一个 solution 删 project 流程上线 / storage 告警关联到孤儿文件。

DEF-03

Metrics counter for materialize_event_failures_total

现状@OnEvent('session.bound') handler 吞错只 logger.warn;跟 SessionAssetSyncer 现有 precedent 一致。

推迟理由项目当前没有 Prometheus / OTel metrics infra。单加这一个 counter 不值得拉起整套 metrics 系统。

重新审视的触发Metrics infra 铺开后,这是第一批要补的 counter(跟 SYSTEM_USER_ID fallback counter 一起)。

DEF-04

Always-on files.materializedCount metadata

现状只在 capped=true 时写一个 {materialized, total, capped:true} 行。

推迟理由没有 UI consumer。Round-2 reviewer 建议 always-on 但承认"当前没读者,defensible";所以留到有需求再扩。

重新审视的触发UI/dashboard 想统计 entity bootstrap 状态(物化几个、跳过几个、失败几个)。

DEF-05

Hot-loop realpath 优化

现状500 文件 cap × 2 次 fs.realpath per file = ~1000 stat calls。

推迟理由实测 sub-second(tmpfs 上),不是瓶颈。Round-2 reviewer 标 "no action required"。

重新审视的触发Bootstrap 时间超 N 秒,或 profiling 显示这是热点。

Open Questions 真正还没答案的

3 项 · 没 confident answer
OQ-01

多 instance ccaas 部署时 in-flight Map 不够

问题D-04 的 inFlight Map 只防同进程 race。多实例部署时,两个 ccaas 实例同时收到同一 session 的 worker job 会重复物化(还可能写穿对方 delta?)。

为什么 openccaas 当前是 single-replica(SQLite + agentfs FUSE 都假设单进程),没有这个 scenario。但多实例路线在 backlog 里;到那一刻需要 DB-level advisory lock,而 agentfs FUSE 那一层根本不能跨 host —— 不止 race 一个问题。等多实例方案明朗再设计。

OQ-02

Cap 命中后 agent 怎么知道有"未物化的文件"?

问题500 文件 / 50 MB cap 超出后,剩下的 entity files 永远不会进 agent 的 fs。agent 不知道它们存在 → 无法主动调 download tool 拿 → 等于功能缺失。

为什么 open可以在 system prompt 里加 hint("还有 N 个文件可以用 download_file tool 拿"),但 system prompt 是 solution 拥有的;平台越界往里写违反 phase 5.5 边界。又或者出一个 ccaas 的 tool list_unmaterialized_files 让 agent 自己查 —— 但 agent 凭什么知道要查?设计还没想透。

OQ-03

多 entity 一文件

问题当前 schema 是单 sourceIdentity。一份课件给多个班/课程共享(教师上传一次,多个课程引用)做不到。

为什么 open需要 join table(agent_file_entity_bindings)+ guard/list 路径都得跟着改。当前没有 solution 报告这个需求,暂不设计 —— 一旦真需求来,会牵涉到 D-05 的"原路径还原"决策(同一份文件在不同 entity 里路径可能要不同?)。
